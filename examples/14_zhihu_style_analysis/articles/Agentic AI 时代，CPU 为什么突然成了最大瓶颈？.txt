最近这半年，我在实际落地一些多 agent 系统的时候，越来越明显地感觉到一件事：

GPU

还是那个风头最盛的明星，但真正卡住整个系统吞吐量的，往往是那些被我们忽视了好几年的服务器

CPU

。

以前聊 AI 基础设施，大家眼睛里只有 GPU——训练看算力，推理看 token/s。可自从 agentic AI 真正开始大规模用起来以后，情况完全变了。Agent 不再是单纯等用户输入再吐个答案，它自己会去规划、调用工具、查内存、协调几十上百个子 agent 并行干活。这些「思考之外」的活儿，全都压在 CPU 头上，而 GPU 经常在那里干等，空转率高得吓人。

我自己前阵子搭的一个分析型 agent 系统，表面上看 GPU 利用率只有 40% 左右，但一查 CPU，核心全在满载，延迟直接拉到秒级。后来我把 CPU 规格往上提了一档，整体吞吐量反而翻了快一倍。这才真正让我意识到：**在 agent 时代，CPU 已经不是「辅助角色」，而是决定整个集群能跑多快的指挥官**。

从纯推理到 Agent 编排，CPU 的角色彻底翻身

先说说过去两年的标准流程：用户发个 prompt，CPU 负责分词，转给 GPU 算，GPU 生成完 token 再扔回 CPU 去解码，最后返回给用户。整个过程里 CPU 就是个「传话筒」，负载很轻，几乎从来不是瓶颈。

现在换成 agent 以后，情况完全不一样了。用户只给一个大目标，agent 先做一次推理生成执行计划，然后拆成一大堆子 agent 并行跑。每个子 agent 可能要调用 API、读数据库、刮网页、做自己的 chain-of-thought……这些子任务完成的时间完全不同，依赖关系也错综复杂。谁先结束、谁等谁、结果怎么合并，全都得 CPU 实时协调。

更要命的是，这种循环会反复进行——做完一轮 reflection，发现没达到目标，就再规划、再推理、再执行。整个过程里，CPU 必须以极低的延迟去监控每个 token、判断子 agent 状态、整合中间结果。GPU 反而成了被喂数据的那个，喂得越快，它才能发挥得越好。

用一句话总结：**纯推理时代是 GPU 决定上限，agent 编排时代是 CPU 决定 GPU 到底能用到几分**。

推理重还是动作重，决定你要多少 CPU

我现在看任何一个 agentic 工作负载，都会先把它放在一条「推理-动作」滑尺上。

推理重的任务，比如让 agent 基于一堆市场报告写一份 2027 年

DRAM

展望分析。它会拆成十几个子领域，每个子 agent 做很长的 chain-of-thought，生成成百上千个 reasoning token，偶尔才调用一次工具。80% 以上的时间其实都在 GPU 上烧算力。这种场景对 CPU 的要求是：单核要够强、跟 GPU 的互联要够快、内存容量要够大，能把超长的 KV cache 塞进去。

动作重的任务就完全相反。比如一个自动执行采购、填表格、发邮件的 agent。它要频繁调用外部 API、读写数据库、处理文件，GPU 反而只在偶尔做决策的时候才用得上。这种时候 CPU 的核心数、I/O 带宽、内存带宽就成了绝对主角。

所以你会发现，同样的一个服务器机柜，以前可能是 8 张 GPU 配 2 颗 CPU，现在很可能要反过来——CPU 的占比大幅上升。NVIDIA 最近高调推出

Vera CPU

能单独部署跑 agent，CoreWeave 直接买来当主力，就是在用实际行动告诉大家：CPU:GPU 的比例要变天了。

怎么挑 CPU？我现在看这九个指标

我自己总结了一套简单的心法，用九个维度去打分，就能大概判断一颗服务器 CPU 适不适合某个 agentic 场景：

- 单核性能和主频（决策要快，监控要实时）

- 核心数量（能同时伺候多少个 agent）

- CPU 到 GPU 的互联带宽（喂数据不能卡）

- 内存带宽和容量（KV cache、上下文越大越好）

- 缓存设计（L3 越大，随机访问越不怕）

- 每瓦性能（几千个 agent 一起跑，电费不是小数）

- PCIe 代数和通道数（工具调用、存储、网络都要快）

- ISA（x86 还是 ARM，生态和功耗差很远）

- NUMA 架构（多 chiplet 时代，不同核心访问内存延迟不一样）

推理重的任务，互联带宽和内存容量几乎是决定性因素；动作重的任务，核心数和 I/O 能力直接决定上限。而所有场景都离不开低延迟和高并发的底层能力。

让我们举一个推理密集型的例子，它侧重于较少的智能体，但每个智能体都要进行扩展的思维链推理，这使得它对 GPU 的要求很高

用户要求Agent基于提供的大量市场调研报告，生成一份关于 2027 年 DRAM 市场前景的全面分析报告。协调器规划了 10 个子研究领域。每个子Agent运行一个冗长的推理循环，在综合分析其负责领域时会生成成百上千个推理标记。子Agent偶尔会检索文档或进行网络搜索，但 80%以上的计算时间都用于推理。Agent进行的是思考而非执行，所有子代理完成后，最终的反射过程会将所有内容综合成输出结果。

表格显示了每个指标的相对重要性及其原因

指标 (Metric)

重要性 (Importance)

原因 (Reason)

单核性能 (Per-core performance)

高 (High)

当 GPU 完成一轮计算后，CPU 必须处理输出、决定下一步并设置下一次推理调用。推理调用之间每一微秒的 CPU 延迟都会导致 GPU 空闲。

核心数量 (Core count)

低 (Low)

场景中可能只有数十个子智能体，而非数百个。每个智能体大部分时间在等待 GPU 推理，而非执行 CPU 任务。

CPU-xPU 互联带宽 (CPU-xPU interconnect BW)

关键 (Critical)

长文本推理会导致 KV 缓存被逐出（Eviction）至 DRAM，随后需要由 xPU 快速访问这些数据。

内存带宽与容量 (Memory BW and capacity)

高 (High)

当长文本场景下的 KV 缓存从 HBM 卸载时，CPU 与 DRAM 之间的内存带宽至关重要。高容量可以防止数据被进一步逐出到更慢的存储介质。

缓存设计/容量 (Cache design/size)

中 (Medium)

缓存是加分项，但非必需，因为数据主要在 CPU 和 GPU 之间流动。

能效比 (Performance / watt)

低 (Low)

CPU 利用率较低，大规模的 GPU 使用占据了绝大部分功耗。

PCIe 版本与通道数 (PCIe gen and lane count)

中 (Medium)

如果上下文过大，以至于 KV 缓存需要存储在本地 SSD 上，那么快速的数据访问就变得很重要。缓存逐出到上下文内存存储将依赖于 DPU。

指令集架构 (ISA)

低 (Low)

工具支持不需要非常广泛。只要 ARM 和 x86 都支持，指令集架构本身不会有太大区别。

NUMA 延迟 (NUMA latency)

低 (Low)

使用的核心并不多。所有核心都可以被固定（Pinned）在同一个 NUMA 节点中。

拿 NVIDIA 的 Vera 举例，它 88 个自定义 ARM 核心，

NVLink-C2C

互联带宽高达 1.8 TB/s，双向一致性内存共享，1.5 TB LPDDR5……这些规格放在推理重的工作负载里，几乎每一项都拉满。难怪它能单独拿出来卖。

写在最后

我以前也跟大多数人一样，觉得 CPU 在 AI 时代就是个配角。结果真正把 agent 系统跑起来以后才发现，

GPU 抢走了所有聚光灯，但真正决定系统能跑多稳、多快的，其实是那些默默干活的 CPU

。

这个转变才刚刚开始。Intel 在最近的财报电话会上都被 CPU 需求打了个措手不及，就知道我们还处于非常早的阶段。未来几年，数据中心里 CPU 的采购量大概率会迎来一波意想不到的爆发。
