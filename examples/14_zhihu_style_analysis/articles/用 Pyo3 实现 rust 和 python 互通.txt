PyGILState_Ensure 是确认 GIL 是否可以调用 Python C API 。 这个和 CPython 的 GIL 线程处理机制有关。关于这个，Python 官网有

When threads are created using the dedicated Python APIs (such as the

threading

module), a thread state is automatically associated to them and the code showed above is therefore correct. However, when threads are created from C (for example by a third-party library with its own thread management), they don’t hold the GIL, nor is there a thread state structure for them.

If you need to call Python code from these threads (often this will be part of a callback API provided by the aforementioned third-party library), you must first register these threads with the interpreter by creating a thread state data structure, then acquiring the GIL, and finally storing their thread state pointer, before you can start using the Python/C API. When you are done, you should reset the thread state pointer, release the GIL, and finally free the thread state data structure.

当通过 Python API 创建线程的时候，线程状态是可以关联的。但是如果通过 C 或者其他第三方库创建线程，就没有通过 GIL，在 Python 侧就没有线程状态管理

如果要通过这类线程调用 Python ，必须先注册这些线程。这样 GIL 解释器就能创建线程状态、获取 GIL 并且最终存储线程状态。

也就是说，我们通过 FFI 直接调用 CPython 的接口的时候，先要获取 GIL。

Pyo3 内部还有一个

ReferencePool

来管理还没被 GIL 获取的对象 (PyObject)。这里面涉及到 Python 的 Ownershop 规则，也就是所有权规则。

当一个 Object 传到一个函数里，如果是传递指针引用，则这个指针也获得了这个 Object 的所有权。当一个对象引用传到另外一个函数，这个函数就 borrow 这个引用。看到这里，是不是感觉和 Rust 的

ownership

也很像，但是 Rust 更严格和复杂一些，有 Move 、Borrow、Cell等。

在 Python 里，如果一个 Object 被函数引用。则这个应用的生命周期在这个函数内部是被授权可以操作对应的 Object。但是需要用 Py_INCREF 记录一下这个应用，在使用完成之后，需要 Py_DECREF 减少这个引用。

GILPool 来管理应用对象的情况，这是一个

RAII

形式的操作，初始化获取资源，Drop 注销资源。

这部分和 CPython 的机制非常相关，只是在 Pyo3 这一侧做了记录和管理。

int

The start symbol from the Python grammar for isolated expressions

int

The start symbol from the Python grammar for sequences of statements as read from a file or other source

The start symbol from the Python grammar for a single statement
