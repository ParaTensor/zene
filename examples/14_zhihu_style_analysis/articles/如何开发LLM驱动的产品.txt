搬砖并翻译自：Building LLM-powered products在过去的 6 个月里，我一直在使用 GPT 和其他人工智能即服务提供商开发由 LLM 支持的应用程序。在此过程中，我制作了一组插图来帮助可视化和解释一些一般的建筑概念。以下是第一批，希望以后能补充更多。1. 基本提示最基本的LLM概念：您向模型发送一段文本（称为提示词 prompt），它会用另一段文本进行响应（通常称为 completion）。要记住， LLM 只是尝试统计最可能的下一个单词。因此，术语“completion”。这是理解LLM如何使用的关键。2. 动态提示词LLM 应用程序中的常见做法是创建一个“提示词模板”，并在将最终提示发送给 LLM 之前动态地用用户数据替换其中的部分内容。3. 提示链接在某些情况下，一次 LLM 交互可能还不够。也许是因为任务很复杂，或者完整的响应不适合上下文窗口（模型可以根据请求读取和写入的最大标记/单词）。这时您可以使用提示链接：将第一个调用的响应合并到下一个调用的提示词中。4. 无幻觉问答（RAG）由于 LLM 本能地回答问题，他们有时会给出不正确的答案，称为“幻觉”。对于客户支持等与事实相关的用例来说，这是一个问题。然而，虽然 LLM 不擅长处理事实，但他们擅长数据提取 and 重新措辞。减轻幻觉的解决方案是为模型提供多种资源，并要求它使用这些资源找到答案或回答“我不知道”。检索资源以增强提示并提高输出的技术称为RAG（检索增强生成）。5.向量数据库使用传统数据库和字对字匹配来检索上述用例的相关资源可能具有挑战性。同一个词在不同的上下文中可以有不同的含义。在向量数据库中，项目（单词、句子或完整文档）通过代表其含义的数字序列进行索引。这些数字是多维空间（通常有 1500 多个维度）中的坐标。其中一些维度可以捕捉诸如“自然←→合成”、“正←→负”、“多彩←→平淡”或“圆形←→尖”等概念。然后，使用几何学，可以轻松计算两个项目的含义之间的距离，并根据我们的输入找到最接近的匹配项。6. 聊天提示由 OpenAI 及其流行的GPT-3.5-turbo模型引入，聊天提示使用稍微不同的格式来表示讨论，其中完成将是序列中的下一条消息。☝ 需要注意的是，虽然 GPT 的聊天 API 需要一系列角色/消息，并使用此特定结构进行微调（优化），但聊天提示相当于格式如下的文本提示：
SYSTEM：您是 AI 客户支持代理... 
AI：您好，今天需要什么帮助吗？
用户：我的互联网连接速度很慢
AI：
☝ 此外，由于 OpenAI 的聊天模型比 non-聊天模型性能更高且成本更低，因此我们经常使用聊天模型来执行不涉及讨论的任务。您可以将指令放在第一条系统消息中，并在第一条 AI 消息中得到您的回复。对于下面的所有示例，您可以互换使用聊天提示和聊天格式的文本提示。7. 聊天对话LLM 是无状态的，这意味着它们不存储先前调用的数据，并将每个请求视为唯一的。然而，当你与人工智能聊天时，它需要知道你最后一个问题之前所说的内容。为了解决这个问题，您可以将聊天历史记录存储在您的应用程序中，并将其包含在每个请求中。8. 压缩提示词LLM 对它们可以在称为“上下文窗口”的单个请求中读取和写入的令牌数量（≈ 字）有限制。例如，如果窗口大小为 4096 个令牌的模型收到 3900 个令牌的提示，则其完成将限制为 196 个令牌。为了避免达到此限制并降低 API 成本（因为提供商根据输入/输出令牌收费），解决方案是使用中间提示来汇总旧消息。  因为这个总结会花费你第二次 LLM 请求，所以在经常做或不够做之间存在一个弱点。☝ 还要知道更大的上下文窗口很少是解决方案。具有更大上下文窗口的模型更昂贵，更大的请求更慢，并且太多的上下文可能会适得其反。9. 概述在我们 (1) 准备好提示并 (2) 将其发送到 LLM 以生成响应后，(3) 我们的应用程序可以读取完成信息以确定后续操作，例如触发事件、执行函数、调用数据库或提出其他 LLM 请求。基本的客户端/服务器人工智能聊天工作流程。10.  JSON 格式化为了根据 LLM 的响应以编程方式触发操作，我们不想玩猜谜游戏，试图通过完成中分散的单词和短语来辨别意图。更可靠的解决方案是提示模型以JSON进行响应，这是一种广泛使用的文本格式，用于表示结构化数据，受到所有主要编程语言的支持，并且是人类可读的。
