描述了多文档 agents 的的一个过程，详细的代码在

LlamaIndex doc

主要步骤是这样 ：

为每个文档创建一个 "向量索引" (VectorIndex) 和 “总结索引”(SummaryIndex)

然后为这两个索引都创建一个查询引擎

然后把这两个索引包装为 Tools，传给

OpenAI

Agent

然后包装一个 Top-Level 的 Agent，也就是包裹了这些 Agent 的总 Agent 。它主要作用是把问题分类提交给不同的 Agent，然后搜集子 Agent 反馈综合返回

看上去是不是很简单，下面有更详细的一个图描述这个过程：

从上图可以看出，关于文档的处理和查询已经发展的比较深入了。不知道这样做是不是可以很好的弥补大模型的“幻觉”问题

LlamaIndex 的作者，在针对 OpenAI 的研究员

Andrej Karpathy

的总结是这么说的：

大模型更完整的画面出现：它不是一个聊天机器人，而是一个新的操作系统。包括：

- 跨模式的输入和输出（文本、音频、视觉）

- 代码解释器，编写和运行程序的能力

- 浏览器/互联网访问

- 用于文件和内存存储和检索的嵌入数据库

将大模型看做聊天机器人就像将早期计算机看做计算器一样，我们会看到一种全新的计算范式的出现，当然现在还为时过早

---- Andrej Karpathy

LlamaIndex 作者 Jerry Liu 回应

我确实认为“RAG 作为聊天机器人”是支持 LLM 的应用程序的更广泛方案中的临时状态。

LlamaIndex/GPT Index 的最初动机是一个更宏伟的愿景，即将 LLM 视为神经 CPU，并通过“神经记忆”来进行编排。 也就是说，聊天机器人仍然有很多价值可以挖掘。如今大多数构建 RAG 的企业都只是在做 top-k。通过大模型进行出色/高性能/高效的搜索/检索仍然是一个巨大的增值。

---- Jerry Liu

RAG 叫做 Retrieval-Augmented Generation，叫增强型的搜索生成。就是对传统的大模型“提问/回答”模式进行增强。因为如果提问的知识大模型并没有或者缺少一些的话，大模型就会产生"幻觉"，也就是会说一些不存在的内容

RAG 就是在提问的时候，去查询一下补充大模型没有的真实的最新信息和特定领域的知识信息等。将这些信息和提问合并之后，再提交给大模型。从而减少大模型产生幻觉的概率

以上的多文档查询就是 RAG 多层使用的一个结果。 可以在提交给大模型前补充数据，也可以回答之后缺少数据的之后通过 Agent 再去获取数据都可以。RAG + Agent + Finetune 构成了目前大模型落地的一个最具性价比的方案

我尝试了一下，在基础大模型上增加外部的数据来做查询，效果的确比没有增加这些内容的准确：

如果直接问 ChatGPT，会回答没有这些内容。问 Google Bard，回答的不错，但是数据来源主要是互联网，没有这些核实过的数据来源准确

随着这个思路发现，RAG 还有更多的一些玩法出现。例如分片(node)、父子节点、元数据增强、多路召回、索引、效果评估(rerank)。详细的可以看这一篇：

构建产品级的 RAG 应用

但是 RAG 技术还面临一个比较大的挑战，如何知道从外部获取的数据和提示词相关。如果获取了不相关、甚至错误的内容，注入到提示词中，会造成召回污染，导致更差的查询结果

向量数据库的不精确性也会影响 RAG 的效果。向量查询是一种近似度查询机制，根据余弦相似度查询最接近的向量。这样在一些要求比较准确的场景就会产生一定程度的误差

所以结合现有的已经处理过的高质量数据进行 RAG 会有更好的效果。例如通过 Agent Tools 通过 API 调用现有数据库中的数据，或者查询返回的结果格式化之后通过一定的验证机制来识别错误。当然，这些就要具体到用户场景里才会有更好的效果。

RAG 技术还在发展，但是它已经预示了数据处理的未来。就是非结构数据和结构化数据混合的场景下，为业务提供灵活而准确的数据支持
