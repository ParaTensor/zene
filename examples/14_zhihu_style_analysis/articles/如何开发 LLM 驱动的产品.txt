Building LLM-powered products

在过去的 6 个月里，我一直在使用 GPT 和其他人工智能即服务提供商开发由 LLM 支持的应用程序。在此过程中，我制作了一组插图来帮助

可视化

和解释一些一般的建筑概念。

以下是第一批，希望以后能补充更多。

1. 基本提示

最基本的LLM概念：

您向模型发送一段文本（称为提示词

prompt

），

它会用另一段文本进行响应（通常称为

completion

）。

要记住， LLM 只是尝试统计最可能的下一个单词。因此，术语“completion”。这是理解LLM如何使用的关键。

2. 动态提示词

LLM 应用程序中的常见做法是创建一个“提示词模板”，并在将最终提示发送给 LLM 之前动态地用用户数据替换其中的部分内容。

3. 提示链接

在某些情况下，一次 LLM 交互可能还不够。也许是因为任务很复杂，或者完整的响应不适合上下文窗口（模型可以根据请求读取和写入的最大标记/单词）。

这时您可以使用提示链接：将第一个调用的响应合并到下一个调用的提示词中。

4. 无幻觉问答（RAG）

由于 LLM 本能地

回答问题，他们有时会给出不正确的答案，称为“

幻觉

”。对于客户支持等与事实相关的用例来说，这是一个问题。

然而，

虽然 LLM 不擅长处理事实，但他们擅长

数据提取

和重新措辞。

减轻幻觉的解决方案是为模型提供多种资源，并要求它使用这些资源找到答案或回答“我不知道”。

检索资源以增强提示并提高输出的技术称为

RAG（检索增强生成）

。

5.向量数据库

使用传统数据库和字对字匹配来检索上述用例的相关资源可能具有挑战性。同一个词在不同的上下文中可以有不同的含义。

在

向量数据库

中，项目（单词、句子或完整文档）

通过代表其含义的数字序列进行索引

。

这些数字是

多维空间

（通常有 1500 多个维度）中的坐标。其中一些维度可以捕捉诸如“自然←→合成”、“正←→负”、“多彩←→平淡”或“圆形←→尖”等概念。

然后，使用

几何学

，可以轻松计算两个项目的含义之间的距离，并根据我们的输入找到最接近的匹配项。

6. 聊天提示

由 OpenAI 及其流行的

GPT-3.5-turbo

模型引入，聊天提示使用稍微不同的格式来表示讨论，其中完成将是序列中的下一条消息。

☝ 需要注意的是，虽然 GPT 的聊天 API 需要一系列角色/消息，并使用此特定结构进行微调（优化），但

聊天提示

相当于格式如下的文本提示

：

SYSTEM

：您是

AI

客户支持代理...

AI

：您好，今天需要什么帮助吗？
用户：我的互联网连接速度很慢

AI

：

☝ 此外，由于 OpenAI 的聊天模型比非聊天模型性能更高且成本更低，因此我们经常使用聊天模型来执行不涉及讨论的任务。您可以将指令放在第一条

系统

消息中，并在第一条 AI 消息中得到您的回复。

对于下面的所有示例，

您可以互换使用聊天提示和聊天格式的文本提示

。

7. 聊天对话

LLM 是无状态的，这意味着它们不存储先前调用的数据，并将每个请求视为唯一的。然而，当你与人工智能聊天时，它需要知道你最后一个问题之前所说的内容。

为了解决这个问题，您可以将聊天历史记录存储在您的应用程序中，并将其包含在每个请求中。

8. 压缩提示词

LLM 对它们可以在称为“上下文窗口”的单个请求中读取和写入的令牌数量（≈ 字）有限制。

例如，如果窗口大小为 4096 个令牌的模型收到 3900 个令牌的提示，则其完成将限制为 196 个令牌。

为了避免达到此限制并降低 API 成本（因为

提供商根据输入/输出令牌收费

），解决方案是使用中间提示来汇总旧消息。

因为这个总结会花费你第二次 LLM 请求，所以在经常做或不够做之间存在一个弱点。

☝ 还要知道

更大的上下文窗口很少是解决方案

。具有更大上下文窗口的模型更昂贵，更大的请求更慢，并且

太多的上下文可能会适得其反

。

9. 概述

在我们 (1) 准备好提示并 (2) 将其发送到 LLM 以生成响应后，(3) 我们的应用程序可以

读取完成信息以确定后续操作

，例如触发事件、执行函数、调用数据库或提出其他 LLM 请求。

基本的客户端/服务器人工智能聊天工作流程。

10.  JSON 格式化

为了根据 LLM 的响应以编程方式触发操作，我们不想玩猜谜游戏，试图通过完成中分散的单词和短语来辨别意图。

更可靠的解决方案是

提示模型以

JSON

进行响应，这是一种广泛使用的文本格式，用于表示

结构化数据

，受到所有主要

编程语言

的支持，并且是人类可读的。

以下是描述旅行标准的 JSON 示例：

{

"departureCity"

"美国纽约"

,

2

:

false

,

"specialRequests"

:

[

"可欣赏埃菲尔铁塔景观的房间"

,

]

,

"min"

4000

描述我们想要的结构

让LLM 自己随机设计

数据结构

是不可靠的。我们的提示必须

描述我们正在寻找的内容。

为此，即使我们想要 JSON 输出，GPT 的常见做法是使用

Typescript 样式的类型定义

。一种简单的语法，可以描述数据类型（文本字符串、数字、

布尔值

等）、可选属性（使用

?

）和指导 LLM 的注释（以 开头

//

）。

地址提取提示示例。

11. 格式输出的一致性

LLM 令人惊叹的非确定性能力是有代价的：它们并不总是遵循格式说明。  有几个因素在起作用：

模型的代码流畅度

模型

最初训练的

代码越少，它可能需要的指令就越冗长。如果它根本没有学到任何东西，我认为你无法通过提示来弥补这一点。

微调

另一方面，如果您仅使用您正在寻找的 JSON 结构示例来微调 LLM，

它将学习完成此任务

，并且不再需要格式指令。

提示词工程

使用 GPT 等足够强大的模型，输出一致性仍然会受到以下因素的影响：

指令

的

措辞

指令在提示中

的

位置

以及

其余的提示/聊天历史记录

，因为它们可能包含混淆模型的句子。

GPT 提示：

使用大型提示时，

请将格式说明放在末尾

，靠近预期响应。

要求模型将

JSON 包装

```json

在和`````

Markdown 分隔符

之间，这似乎有助于它编写有效的代码并方便提取。

从LangChain

的

结构化输出解析器

等工具中获取灵感，但确保它适合您的特定需求和您使用的模型。

12.函数调用

让我们回到我们的应用程序！

现在，如果我们用可能的操作列表

提示模型并请求 JSON 格式的完成，我们的应用程序可以轻松解释它并

为每个操作执行相应的代码

。

这使得人工智能能够与我们应用程序的各个组件（内存、数据库、API 调用等）进行交互。

安全问题

显然，如果不受控制，让非确定性人工智能在您的系统中运行会让您面临安全问题，例如

提示注入

（黑客精心制作一条消息，让模型执行他们想要的操作）。为了缓解这种情况：

避免让 AI 执行

任意代码

，例如运行其自己的 SQL 查询。相反，将其交互限制为一组预先批准的操作。

使用用户级权限执行 AI 操作，

以确保它无法访问其他用户的数据。

在使用人工智能工具之前，*

*请确保您了解它们

[

的功能**](

https://

positive.security/blog/

auto-gpt-rce

)。工程专业知识和

批判性思维

在这里至关重要。

OPENAI的函数调用

2023 年 6 月，OpenAI 发布了对其 GPT

函数调用功能的访问，该功能与

聊天提示

非常相似，引入了一种新的专有 API 语法，

抽象了提示可以完成的操作

，并对其模型进行了微调以提高可靠性。

☝ 它们的实现是可靠的，但尚未成为

人工智能模型

和提供商的标准。就像 LangChain 一样，它代表了易用性和对技术的控制之间的权衡。

13.工具（插件）

结合 JSON 响应、函数调用和

提示链

，我们可以组装一个能够

完成涉及外部资源的任务

的智能助手。

日历管理器 AI 助手示例。

我们首先运行第一个 LLM 查询，为模型提供

任务和一组工具

。

根据其响应，我们执行相应的函数，并在后续查询中将

结果反馈给模型。

我们重复这个过程，让人工智能决定下一步，

直到得出最终答案

。
