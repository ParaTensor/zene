在这次

大语言模型

的快速发展中，langchain 作为一个很独特的产品。在发展之初被一些观点认为是第一波被替代的产品。在大家逐渐熟悉之后，又被认为太简单，技术上没有门槛，非常容易被替代。

在最新的 a16z 的

Emerging Architectures for LLM Applications

文章中，

langchain

被定义为 LLM 的编排工具

那么 langchain 到底是什么呢，在 LLM 应用中可以用来做什么呢？

从我个人的开发体会来看，langchain 是使用现有的技术栈来更有效的和 LLM 交流的一套方法，体现为一套工作流和接口。所以理解为 LLM 编排也是准确的

上面这个说法可能还是有些抽象，不容易理解

我们以前和系统的交互方式，会通过 API 或者一些界面操作，把要做的东西提交给后台，得到返回。现在 LLM 是后台，LLM 是语言模型，遵从一些语言模型的交互方式。就是所谓”提示词“(prompt)

提示词是提交给 LLM 的一段话，而 LLM 如何理解这段话，然后理解之后采用合适的动作来执行一些任务，返回正确的内容。就是我们和 LLM 交互的一个流程

而 LLM 阅读提示词然后执行的这个机制，有几种。就是所谓

zero-shot

、one-shot、

few-shot

、ReAct等等zero-shot 叫做无样本学习，就是你直接问问题，然后直接回答。不需要有一个其他的学习过程

one-shot 和 few-shot，就是有样本学习，一个或者多个。你需要预先设定一个内容，作为样本，提交给 LLM，然后获得回答

ReAct 相对复杂，就是在提交的过程中，可以和 LLM 反馈之后再次输入交互。这也是 AutoGPT 的运作方式。

随着研究深入以及 LLM 的多样化和演进，会有更多的和 LLM 交互的方法提出。langchain 就为了简化交互方式，提出了一个 "chain" 的概念来抽象和 LLM 的交互方式

在 langchain 文章这样描述：

LangChain provides the

Chain

interface for such "chained" applications. We define a Chain very generically as a sequence of calls to components, which can include other chains.

简单的 LLM 应用可以是单独运行的，不和其他系统交互。但是更多复杂的 LLM 应用需要调用其他 LLM 应用，以及其他组件。所以 langchain 提供了一个 Chain 的接口来表示这种有 “链”的应用。定义 chain 为组件的调用序列。

理解下来，chain 就是一堆 LLM 程序的次序调用，也可以支持

嵌套调用

。这个比较容易理解，因为在我们其他的应用程序，实际上也是这样。一个应用程序完成一个功能，把结果提交给下一个应用。也可以来回调用和嵌套调用。

除了 chain，langchain 还定义了其他概念：Model I/O、Data connection、Memory、Agents、Callbacks

Data connection 比较容易理解，但是其实非常重要。langchain 通过 data connection 定义了 llm 的

数据处理

过程：

Agents 最近有点时髦，因为 OpenAI 也在提 AI Agent。Agent 可以理解为针对用户的完成一组功能的 chains。提供了和用户输入交互的能力，根据用户输入调用不同的 chains 和其他组件来完成任务

Memory 为 chain 和 agent 提供记忆能力。因为 chain 和 agent 是无状态的，如果做一个复杂的，交互比较多，时间比较长的任务，需要记录中间的状态，就需要用 memory 来记录输入输入的历史

Callback 是为了能够获取 LLM 操作后的延迟反馈。也是应用系统的一个常规设计

我们再看一下 langchain 的定义：

langchain 是使用现有的技术栈来更有效的和 LLM 交流的一套方法，体现为一套工作流和接口。理解为 LLM 编排也是准确的
