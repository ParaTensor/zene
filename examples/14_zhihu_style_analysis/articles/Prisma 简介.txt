To answer the question briefly:

No, Prisma is not an ORM

.

ORMs are libraries that map tables in your database to classes in your programming language. Prisma, on the other hand, is a database toolkit. The toolkit includes Prisma Client, which is an auto-generated query builder that exposes queries which are

tailored

to your models. All Prisma Client queries return plain old JavaScript objects.

ORM 是

映射数据库

的表到编程语言上的类。而 Prisma 是一个数据库工具，能够根据

数据模型

生成特定的查询工具。这句话听上去有些绕，经过一段时间的体验，有了一些理解

Prisma 会根据数据库的 Schema，生成 javascript 的客户端库。在里面定义了相应的数据接口，关联关系和函数等。通过这些，来完成对数据的操作

这里面要了解 schema 的概念，还有通过 schema 建立的数据关联关系，以及如何来查询和新增数据等

在 Folder 里面定义了 Files:

model Folder {
  ...
  files           File[]  
}

这些都是通过

FOREIGN KEY ("folderId") REFERENCES "public"."Folder"(id)

这个 SQL 语句通过 prisma introspect 自动生成的。这样就定义了一个从 Folder 到 File 的一个 1:n 的关系。

在自动生成的时候，Folder 是这样的

File File[]

。需要更改为 files，小写，后面加 s，代表是多个。

如果需要生成 1:1 的关系，只需要在对应的 folderId 增加一个 UNIQUE 的标记，例如：

CREATE TABLE "public"."File"(
	...
  "folderId" INTEGER NOT NULL UNIQUE,
  FOREIGN KEY ("folderId") REFERENCES "public"."Folder"(id)
);

这样生成的 Folder 就会是

File File[]

，然后改为

file File[]

对应关系

1:1 和 1: N 前面描述过了。还剩下 N:N 和

自关联

两种

M:N 需要增加一个关联表，官网上有

描述

CREATE TABLE "Category" (
    id SERIAL PRIMARY KEY
);
CREATE TABLE "Post" (
    id SERIAL PRIMARY KEY
);
-- Relation table + indexes -------------------------------------------------------
CREATE TABLE "_CategoryToPost" (
    "A" integer NOT NULL REFERENCES "Category"(id),
    "B" integer NOT NULL REFERENCES "Post"(id)
);
CREATE UNIQUE INDEX "_CategoryToPost_AB_unique" ON "_CategoryToPost"("A" int4_ops,"B" int4_ops);
CREATE INDEX "_CategoryToPost_B_index" ON "_CategoryToPost"("B" int4_ops);

我们在这里构建了一个 Folder 树，要用到 self-relations，在 SQL 这样定义:

CREATE TABLE "public"."Folder" (
  id SERIAL PRIMARY KEY NOT NULL,
  "projectId" INTEGER NOT NULL,
  name VARCHAR (255),
  level INTEGER,
  "parentFolderId" INTEGER UNIQUE,
  UNIQUE (level, name),
  FOREIGN KEY ("projectId") REFERENCES "public"."Project"(id)
);
ALTER TABLE "public"."Folder" ADD CONSTRAINT fk_parent_folder FOREIGN KEY ("parentFolderId") REFERENCES "public"."Folder" (id)

parentFolderId 是 Unique ，因为子目录只能有一个父目录。parentFolderId 也是 Folder 自己的外键。生成的 schema.prisma 是这样：

model Folder {
  id              Int      @default(autoincrement()) @id
  ...
  parentFolderId  Int?
  parentFolder    Folder?  @relation("FolderToFolder_parentFolderId", fields: 	[parentFolderId], references: [id])
  other_Folder Folder[] @relation("FolderToFolder_parentFolderId")
}

生成了一个 other_Folder，other_Folder 和 parentFolder 的 relation 名字是一样的，叫做"FolderToFolder_parentFolderId"；parentFolder 包含针对 parentFolderId 的 references

other_Folder 就比较费解，在官方文档里也没说明。我的理解，other_Folder 就是 childrenFolders，所以改名为childrenFolders
