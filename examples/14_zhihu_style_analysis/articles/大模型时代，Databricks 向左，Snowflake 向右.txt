最近在看

Databricks

和

Snowflake

的产品路径。发现这两家在大数据时代的领先者，在大模型时代有着显著不同的策略

Databricks 的方案以 RAG(增强搜索)为核心，如下：

Image.png

用户输入问题，通过从事先处理好的文本向量索引里面获取问题相关信息，加上提示词工程，生成回答。上图上面的 RAG Chain 就是主要流程。下面左边的

Unstructured Data Pipelines

是事先的文本切分、向量化和索引的过程。这都是典型的 RAG 操作流程

但是在下面右边有一个

Structured Data Pipelines

是 Databricks 特征工程处理流程。那这个结构处理在 RAG 流程中有什么作用呢

Image.png

从文章上没看出有什么具体作用。 Databricks 为什么要硬塞一个 Structured Data Pipeline 在 RAG 流程里呢，这可能和 Databricks 的定位有关系。

Image.png

Databricks 在大模型时代，描述自己是数据智能平台。这个平台的底座还是 databricks 最核心的产品

Delta Lake

数据库。传统的SQL/Table数据处理流程是 databricks 在大数据时代的制胜法宝。之后 databricks 针对上一代深度学习推出了

MLFlow

、

Feature Store

，反响一般。这次在大模型时代，Databricks 收购了

MosaicML

，这实际上是一条独立的路线，和 databricks 传统的 Delta Lake 及 MLFlow 没有太多关联。

Databricks 定位为数据平台，势必是从数据角度来看大模型的发展趋势。大模型在 RAG 方向上发展迅猛。 RAG 有一个核心环节是召回（Retriever），用来补全问题的背景信息，提高查询准确度。这个召回按照目前的做法是从向量数据库召回，但是向量也存在不准确性，所以大家就会想着从原来的准确度较高的数据存储中召回。所以 databricks 可能认为，原来的 feature store 或者 table 就派上用场了。或者说，也是为了为了带货

Databricks 描述自己的定位这样说：

数据智能平台背后的核心理念 我们称之为数据智能。它们建立在 Lakehouse 的基础上——一个查询和管理整个企业所有数据的统一系统——但会自动分析数据（内容和元数据）及其使用方式（查询、报告、沿袭等）以添加新的能力 数据智能平台能够：自然语言访问、语义编目和发现、自动化管理和优化、增强治理和隐私、对 AI 工作负载的一流支持：

因此，Databricks 在大模型领域的战略举措是，借助具有广泛市场需求的RAG应用，将自身强大的Lakehouse数据处理能力与上一代深度学习技术栈深度融合，构建出一体化解决方案，并向客户推广这一统一服务

那 Snowflake 是怎么设计自己的大模型策略的呢？

Image.png

Snowflake 的产品结构非常清晰。以Snowflake 数据层为基础，上面提供大模型的容器服务和

Snowflake Cortex

服务。 Cortex 是Snowflake AI 能力产品的核心，这个等会详细描述。在 Cortex 和大模型容器服务至上，构建两个服务，一个是 Snowflake 原生 AI 应用，或者叫函数功能，另外一个就是定制化 AI 应用，这个和通常的路径差不多

Corex 提供了一个 serverless 计算能力之上的功能，包括总结、翻译、预测、分类等。还提供了两个通用能力，Text2SQL 和生成能力。生成能力就是基于大模型的生成式 AI 的封装。上面提供

Document AI

、搜索和 Copilot 原生应用。

这里面有两个比较有意思，

text2sql

和 document AI

还有一个和 databrick 的显著不同，这里没有 RAG。或者说，snowflake 把 RAG 定位为用户的自定义大模型应用，而不是 snowflake 原生能力

text2sql 容易理解，就是自然语言翻译成 sql 。 上面的 Copilot 应用的基础也是 text2sql 。但是 text2sql 目前的技术准确度还较差，为什么 snowflake 把这个作为一个核心的方向呢。

在 snowflake 研究员

一篇文章

中，详细说明了 text2sql 的做法。

Image.png

就是采用执行分数（Execution Score）作为一个评测指标，就是用模型来评估结果。详细的过程可以看原文

Snowflake希望能够在 text2sql 方面达到可用，采用了预测意图和执行分数评估的方法来训练一个 text2sql 模型

Image.png

Document AI 是 snowflake 的从文档直接提取数据的产品。

Image.png

这个产品看上去像是一个比较好用的工具，可以帮助客户单独或者批量从非结构化数据中提取内容

说到这里，databricks 和 snowflake 的大模型策略也比较清晰了。但是实质上，他们都没有触及大模型最新的数据分析路线。就是类似 ChatGPT 的 Advanced Data Analysis，也就是之前的 code interpreter

Image.png

借助 code interpreter 和大模型能力，直接针对非结构化文档做数据分析和可视化

Image.png

这个做法是颠覆性的，实际上也是对 databricks 和 snowflake 技术基础的颠覆。所以他们不会去推动这块技术发展，也是在情理之中

总结一下，现在我们在数据处理技术路径上有 RAG 、 text2sql 和Advanced Data Analysis

databrick、 snowflake 和 openai 分别在发展自己看重的技术。未来 2 年，在数据分析市场应该会有更多好戏可看
