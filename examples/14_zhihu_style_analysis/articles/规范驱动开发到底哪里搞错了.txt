文章正文:
======================================================================

最近我在带团队用 AI 写代码的时候，越来越觉得一件事特别扎心：真正能 100% 信任的文档，其实只有代码本身。那些设计文档、更新日志、README、架构图、 onboarding wiki……几乎写完没多久就过时了。

维护一份书面文档跟上系统变化，是一件持续不断的隐形工作。可工程师天生就是冲刺型动物——写完文档、发版功能、继续下一个任务。更新文档这件事永远排在日程表的最后，而且几乎每次都输给其他更紧急的事。我们试过流程、试过工具、试过把“文档更新”立为团队价值观，可都没用，因为我们一直在要求人类去做一件人类几乎永远做不好的事。

这正是

规范驱动开发

（spec-driven development）最容易翻车的地方。想法本身是好的：跟 coding agent 合作前，先把你要的东西写清楚，总比直接把 prompt 扔进聊天框然后祈祷强多了。

但问题在于，规范本身也是一份文档。而我们刚才已经说清楚文档会遭遇什么命运。

区别只在于代价大小。一份过时的设计文档，最多误导下一个看它的工程师；一份过时的规范，却会误导那些根本不知道更好的 agent。它们会信心满满地执行一个已经跟现实脱节的计划，而且完全不会主动告诉你哪里出了问题。

所以我们开始做

Intent

的时候，反复在想一个问题：能不能让规范不再需要人工维护？能不能让它自己维护自己？

我们最后落地的方案是这样的——规范既不是纯人类文档，也不是纯 agent 文档，而是双方都读、都写的共享活文档。

你先用自然语言描述想做什么，协调 agent 就会根据你的代码库自动起草一份规范，顺便拆成几个子任务。你看一遍，改改细节，觉得 ok 了再点批准。agent 开始干活后，它们会主动把发现的新情况、遇到的约束、做出的调整都写回规范里。你随时可以暂停，改写规范的某一部分，agent 就会从新状态接着干。

这其实很像你把任务交给一个靠谱的初级工程师。你给他一个 ticket，他去干活，发现接口不支持原来 ticket 里假设的分页逻辑，他不会闷头把错的东西做完，而是主动把 ticket 更新了：“这个假设不对，我改成了这样，理由是……”。你 review 他的更新，要么批准，要么 push back。

我们想要的，就是开发者跟规范之间建立这种关系。ticket 之所以能一直保持诚实，是因为双方都在维护它。

这个初级工程师的比喻其实还能往下说。好的初级不会把每一行代码的决策都汇报给你，他只会把那些改变方向的关键决定亮出来：“我发现代码里已经有现成的

auth context

了，所以我直接接进去，而不是新造一个。” 这才是你真正想看到的信号。agent 也一样。把这个粒度调对，其实是整个系统里最有意思的设计挑战之一——太多信息就成了噪音，你学会自动忽略；太少信息你就又得猜到底发生了什么。

举个实际的例子。你写一句：“在设置页加一个深色模式开关，要跟随系统偏好。” 协调 agent 读完你的代码库，起草规范，拆成三个子任务：加开关组件、接偏好存储、更新 CSS 变量。你扫一眼，发现它漏了“跨会话持久化”这一点，就顺手加了一行，然后批准。

十五分钟后，其中一个 agent 把规范更新了：“发现代码库里已有现成的

theme context provider

，直接接进去了，没新建 store。”

你再看它提交的代码改动（按 agent 和任务分组得清清楚楚），规范里现在记录的就是真正被实现的样子，而不是最初计划的样子。而且整个过程没人需要特意记得去更新文档。

软件历史上所有“文档优先”的尝试，最后都因为同一个原因失败了：它们要求开发者去做持续的维护工作，而这份工作没人看得见、也没人奖励。

规范驱动开发如果想不重蹈覆辙，agent 就必须承担起它们该承担的那份维护工作。

它们既然能写代码，就一定也能更新计划。

你最近在用 AI 写代码的时候，是怎么跟规范打交道的？欢迎评论区聊聊，我特别想听听大家真实踩过的坑和找到的解法。
