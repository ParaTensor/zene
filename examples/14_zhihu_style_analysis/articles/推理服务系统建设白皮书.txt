引言

大型语言模型

(LLM) 在各个领域的应用日益广泛，行业和开源社区对其进行了广泛的探索。随着这些模型在生产环境中需求的不断增长，评估其性能以及可用工具和框架的需求变得越来越重要。本白皮书旨在深入探讨大模型模型服务框架，希望为大模型落地提供一些见解，以便在项目中采用合适的技术路线达到更好的效果

推理服务和

推理引擎

大模型推理服务的过程呈现出一个典型的

请求-响应

模型，多个用户同时向服务端发送推理请求，服务端经过一系列处理后将推理结果返回给用户。在这个过程中，服务端通过多引擎后端服务、调度队列和多模型管理等机制，实现了对不同类型推理请求的高效处理和对多种大模型的灵活调度。如下图：

在这个过程中，既需要满足不同用户针对不同模型的快速推理需求，也需要满足整体系统的服务质量。需要推理服务基于推理引擎的能力，并且增加分布式智能调度能能力来达到这个目的

推理引擎

推理引擎是执行推理过程的组件。它负责加载训练好的模型，接收用户输入的内容，并根据模型的逻辑和参数输出结果。推理引擎会针对模型和硬件进行优化，实现低延迟的推理。支持多种模型格式（如 Pytorch、gguf、多模态）和多种硬件（CPU、GPU、NPU 等）

推理服务

推理服务是一个完整的系统。它封装了推理引擎，并且提供了用于模型管理、部署、扩展和监控的接口和工具。推理服务还需要提供多卡和多机条件下的集群管理和调度优化

推理服务需要具备较强的多机异构环境下的调度能力，让整个系统具备高可用、弹性和可配置

衡量推理服务的关键指标

在提供这些模型服务时，有两个重要的指标需要强调：

吞吐量

吞吐量是指推理服务器在处理用户的多个请求过程中每秒生成的 token 数量。吞吐量越高越好。

延迟

延迟表示服务器和模型生成输出序列中完整输出所花费的时间。如果生成的输出是流式传输给最终用户的，则它表示推理服务器生成第一个 token 所花费的时间（也称为 time to first token (TTFT)）。

在构建推理服务平台时，关注吞吐量还是延迟，取决于具体应用场景和需求。

何时侧重吞吐量：

高并发场景：

当您的服务需要同时处理大量用户的请求时，例如聊天机器人、在线翻译等，吞吐量就显得尤为重要。此时，您需要确保系统能够快速响应大量请求，保证用户体验。

批量处理任务：

如果您的应用涉及到批量处理数据，如文本摘要、情感分析等，高吞吐量能够显著提升处理效率。

何时侧重延迟：

实时交互应用：

对于需要实时交互的应用，如语音助手、游戏等，低延迟是至关重要的。用户期望得到即时的反馈，过高的延迟会严重影响用户体验。

对时间敏感的任务：

在一些对时间敏感的任务中，如金融交易、实时决策等，延迟直接影响到结果的准确性和有效性。

往往在一个系统中，即有关注延迟的场景，又有关注吞吐的需求。这就要求推理服务系统具有可配置的能力，同时满足多种场景的不同指标上的要求

推理引擎的关键技术

就像上文所说，推理引擎主要专于提供低延迟的推理。就是尽可能的利用模型和硬件特性来提高推理速度。其中一个非常重要的就是针对 KV Cache 的优化

这里可能就需要花一些篇幅来说明一下大模型推理的一个过程：

在

Transformer

模型中，

自注意力机制

是核心组成部分，它允许模型考虑输入序列中每个位置的token与其他所有位置token的关系。这一过程涉及计算查询（Query）、键（Key）和值（Value）向量。具体来说，模型通过将输入序列的嵌入向量经过线性变换得到Query、Key和Value，然后通过Query与Key的点积计算注意力权重，再用这些权重加权平均对应的Value向量，以获取上下文信息

如上图，输入的 query、key 和 value 分别经过线性变换得到对应的向量表示。然后，query 和 key 进行矩阵乘法，得到相似度得分。接着，将相似度得分进行缩放，并通过 Softmax 函数进行归一化，得到注意力权重。最后，将注意力权重与 value 进行矩阵乘法，得到最终的输出向量。这个输出向量融合了所有 value 的信息，并且每个 value 的贡献由注意力权重决定，从而实现了对输入信息的加权平均。

简单来说，注意力机制就是通过计算 query 和 key 的相似度，来决定对 value 的关注程度，最终得到一个融合了所有 value 信息的输出向量

上述描述的是 Self-Attention 的过程，在 Transformer 中还会用到

Multi-Head Attention

，即并行地进行多次上述过程，然后将结果拼接起来。

这这个计算过程中，计算量最大的部分在于 Q 和 K 的矩阵乘法，用于生成注意力权重矩阵。而 V 参与的计算相对较少，只是与注意力权重矩阵进行一次矩阵乘法。并且对于已经生成的 token，它们对应的 K 和 V 在后续的解码过程中是不变的。因此，可以将这些 K 和 V 缓存起来，避免重复计算，从而显著提高推理速度

在推理过程中，KV 会越来越大。导致占用大量显存。随着上下文长度、用户并发数的增加。KV 可能会占用原始模型的 10% 到数倍的显存。而每次进行下一个 token 的计算都需要访问 kv，导致整体推理的并发能力下降，会极大影响推理速度

所以针对这个问题，出现了很多优化方案。大部分的推理引擎也是针对 kv 做了大量的优化。

PageAttention

PageAttention 通过把 KV Cache 切分为多个小块（或称为 page “页面“），每个页面大小固定。根据实际需要动态分配给不同序列。并且通过映射表记录这些页面，这样在请求的时候就可以有效降低显存脆片和浪费。同时还支持 Copy-on-write 机制，只有在内存需要修改时才复制显存，进一步节省显存

RadixTree

RadixAttention 使用了基数树（Radix Tree）作为数据结构来存储 KV缓 存。基数树允许高效地存储和检索具有相同前缀的键，因为它们在树中共享相同的路径。这种结构使得查找具有相同或相似前缀的键变得非常快速，从而加速了缓存命中和重用的过程

Speculative Execution

Speculative Execution（推测执行）用来预测模型的下一步操作，提前执行可能需要的计算，以减少等待时间。模型基于当前信息预测性地执行后续步骤

优化路径的对比

通过以上的对比可以看出。 vLLM 和 SGLang 这样的推理引擎重要是通过对 kv cache 的优化，以及其他类似推测执行的类似“操作系统”的内存控制和计算调度的技术来优化推理速度。而 TensorRT 是 NVIDIA 的大模型推理引擎，专注于利用 NVIDIA GPU 底层硬件加速和编译优化来提升推理速度

从评测来看， vLLM 和 SGLang 的推理速度已经超过 TensorRT

上图显示 8 张 A100 的服务器上，平均延迟 vLLM 最好。但是在吞吐性能上，SGLang 更加突出：

对于 Llama3-70B-F8 的并发请求，SGLang 保持稳定的性能，在所有请求中始终每秒处理 30-31 个令牌。相比之下，随着请求数量的增加，vLLM 的性能明显下降，从每秒 22 个令牌开始下降到每秒 16 个令牌。这表明 SGLang 更适合处理高并发性，在繁重的工作负载下提供更可靠的吞吐量

推理服务的关键技术

一个完整的推理服务的重要功能是批量处理用户请求。单独为某一个推理请求提高速度在实际的应用场景中还不完全能偶满足需求。实际的场景往往是系统有数量众多的 GPU 或者其他类型的算力芯片(例如 NPU 、TPU)。用户的请求还比较多样。例如针对不同模型的请求，对请求的速度有的是低时延、有的是长上下文。随着模型的多样性，还需要一次处理多个模型的请求，例如针对一个复杂的 RAG 或者 Agent/Workflow 的请求。

先从如何批量处理用户的请求开始

推理的过程分成 prefill (预填充) 和 decode (解码) 两个过程。在预填充阶段，模型会一次性处理所有输入的 tokens， 计算 K 和 V 矩阵。这个阶段计算量比较大，但是只需要做一次。然后把计算结果储存起来供后面的解码过程使用

在解码过程，根据用户的输入和预填充生成的 KV 矩阵，逐次生成可以输出的 tokens。这个过程可以理解为模型“思考”和“表达”的过程，讲理解转换成具体的输出内容。在解码阶段，因为每次只计算一个 token 输出，计算量小但是需要快速输出，对延迟更敏感

如果同时有多个用户同时输入，那如何更好的提高系统整体服务能力呢？

在有多个用户输入的情况下，一般来说也会有多卡，甚至多机。所以这个问题可能更全面的描述是，在多机多卡的资源条件下，如何更好提高推理系统服务能力，来满足多个用户的请求

推理服务系统通过模型并行技术来实现一个模型推理用多张卡，主要方式是

张量并行

。也就是将模型的层或参数分割到多个 GPU 上，每个 GPU 只负责计算模型的一部分。张量就是多维数组的泛化，模型中的参数（例如权重和偏置）和中间激活值都是以张量的形式存储的。

在进行模型切分的时候，如何把模型权重更合理的分配到 GPU 上，以应对未来的并发请求压力，是一个需要权衡的问题

一种办法是将参数一次性加载到 GPU 上，并基于这些参数一次处理尽可能多的输入序列。这种办法可以明显的提高服务器吞吐。例如下图，推理服务会等待一定数量的用户请求，例如图中的 4 个请求 (S1, S2, S3, S4)。积累到足够数量的请求后，服务器会将这些请求打包成一个批次，并将其送入 LLM 模型进行处理。LLM 模型会并行处理批次中的所有请求，例如在时间点 T1，同时处理 S1、S2、S3 和 S4 的第一个 token

但是这样的方法造成了用户的等待时间较长，而且 GPU 利用率低。由于每个请求的长度不同，一些请求可能很快完成 (例如 S3 在 T3 完成)，而其他请求可能需要更长时间 (例如 S2 在 T8 完成)。这会导致 GPU 在某些时间点处于空闲状态，降低了 GPU 利用率。S3 在 T3 就完成了，但由于采用了静态批处理，服务器必须等到 S2 在 T8 完成才能释放整个批次，这导致 S3 的结果被延迟了 5 个时间单位

如何在有限的 GPU 内存下，高效地处理长度不一的请求，并保持高吞吐量呢。

连续批处理（Continuous Batching）

正是为了解决这个问题而出现的一种技术。

相对静态处理，上图的不同在于：在 T6 时刻，S1 完成 (红色方块)，新的请求 S6 加入批次 (蓝色方块)；在 T5 时刻 S3 完成，新的请求 S5 加入批次；在 T7 时刻，S4 完成， 新的请求 S7 加入批次。同时，请求完成后立即从批次中移除，释放内存空间，例如 S1, S3, S4。模型不断处理批次中的请求，并在每个时间步长动态更新批次内容

连续批处理可以缩短等待时间、提高吞吐和优化内存使用。是目前推理服务优化的一项关键技术

如何构建或选择一个好的推理服务平台

一个好的推理服务系统评价的指标包含如下几个：

吞吐量：

平台每秒能够处理的请求数量，直接影响服务容量和用户体验。

延迟：

处理单个请求所需的时间，对低延迟应用非常关键。

可扩展性：

平台能否随着请求量的增长而横向扩展，以满足不断增长的需求。

资源利用率：

平台能否有效利用计算资源 (GPU、CPU、内存)，避免浪费。

因为推理服务技术发展很快，不断会有新的推理引擎出现。所以需要推理平台有能够兼容主流推理引擎的能力。另外因为推理的硬件算力多种多样，需要推理平台能够兼容主流的算力硬件，包括英伟达、华为、AMD 等

选择或者构建 LLM 推理服务平台是一个复杂的过程，需要综合考虑多个因素。建议您仔细评估您的需求和优先级，并对候选平台进行充分的测试和比较，以选择最适合您的平台
